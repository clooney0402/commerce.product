01 — 추진 배경

기존 방식의
두 가지 한계

⏱️
실행 속도 문제
  • 기획 문서 작성에 많은 시간이 소요됨
  • 실제 인터랙션을 이해하기 어려움
  • 디자인 단계에서 재논의 반복 발생
  • 과제 진행 중 정책 확인이 반복됨
  • 결과적으로 과제 착수까지 리드타임 증가
🔄
협업 방식의 한계
  • 기획 의도 전달 오류 빈번
  • 인터랙션 해석 차이 발생
  • 정책 누락으로 QA 단계에서 발견
  • 정의되지 않은 Edge Case 존재
  • 기획 / 디자인 / 개발 수정이 반복됨
🧭
해결 접근 방식
  • 완벽한 기획 이후 착수하는 방식에서 탈피
  • Proto 기반으로 빠르게 설계하고 실행
  • 팀 전체가 함께 과제를 발전시키는 구조
  • 문서가 아닌 동작으로 합의하는 방식
  • AI 워크플로우로 설계 속도 가속

04 — Proto Package

과제 실행을 위한
최소 설계 패키지

01
User Flow
사용자 행동 흐름을 시각적으로 정의. 어떤 경로로 기능에 접근하고 목표를 달성하는지 매핑.
02
Screen Structure
화면 구성 정의. 어떤 요소가 어디에 배치되는지를 와이어프레임 수준으로 구조화.
03
Interaction
버튼과 액션의 동작을 명확히 정의. 클릭 후 어떤 상태가 되는지 구체적으로 기술.
04
Policy
구매 제한, 권한, 예외 처리 등 비즈니스 로직과 정책을 사전에 명확히 정의.
05
Edge Case
예외 상황을 사전 식별하고 처리 방식을 정의. QA 단계 전에 이미 해결책 마련.
✦
Proto가 명확할수록
디자인 이해도 상승, 팀 간 질문 감소, QA 이슈 감소. 실행력은 Proto의 품질에서 시작됩니다.

06 — Lessons Learned

실험에서 얻은
핵심 인사이트

01
🏗️
Proto 설계가 핵심이다
AI 도구보다 중요한 것은 Proto 기반 설계 구조입니다. 도구는 수단일 뿐, 명확한 Proto 없이는 어떤 도구도 효과가 없습니다.
02
🤝
협업 방식의 변화가 필요하다
단계별 전달이 아닌 협업 기반 설계로의 전환. 기획·디자인·개발이 동시에 참여하는 구조가 실행 속도와 품질 모두를 높입니다.
03
🎯
Spec Freeze 기준이 중요하다
언제 스펙을 확정하고 과제를 실행할지에 대한 명확한 기준이 필요합니다. 모호한 기준은 결국 재작업으로 이어집니다.
